# Governance

## Purpose

The Digital Public Infrastructure Ecosystem Atlas is intended to be maintained as a public-interest open source project.

Its governance should favor:

- transparency
- maintainability
- documented decision-making
- structured public data
- accessibility and sustainability as non-optional constraints

---

## Current Governance Model

At this stage, governance is maintainer-led with public contribution pathways.

Current practical controls are in:

- version-controlled schemas and datasets
- pull requests for code and data changes
- public issue tracking
- documented architecture and schema decisions
- monthly automated refreshes with reviewable outputs

This is intentionally lightweight for early implementation while leaving room for a more formal steering structure later.

---

## Decision Priorities

When tradeoffs arise, the project should prioritize:

1. accessibility and public usability
2. data quality and provenance
3. maintainability and transparency
4. interoperability and long-term stewardship
5. performance and sustainability
6. feature expansion

The Atlas should not optimize for novelty, vanity metrics, or dashboard complexity.

---

## Scope Boundaries

The Atlas is not just a catalog of repositories.

It is intended to support:

- ecosystem intelligence
- public infrastructure mapping
- governance maturity analysis
- stewardship analysis
- interoperability analysis
- accessibility and sustainability analysis
- procurement-readiness analysis

This affects contribution review: additions should strengthen ecosystem understanding, not just increase record count.

---

## Contribution Expectations

Contributions should prefer:

- explicit provenance
- stable entity identifiers
- clear rationale for inferred relationships
- reproducible enrichment logic
- accessible UI changes
- minimal unnecessary dependencies

Contributors changing data should avoid making unsupported claims without source or confidence metadata.

---

## Future Governance Evolution

As the project grows, governance may expand toward:

- a small editorial or stewardship group
- documented review standards for data inclusion
- ecosystem-specific maintainers
- accessibility and research review checklists
- a published change policy for schema evolution

A more formal model can be adopted later without blocking phase-one delivery.
